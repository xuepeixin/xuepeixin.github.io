我是怎么搭建一个 Telegram 日志整理 Agent，并把 Topic 工作流接进 Obsidian 的

最近我一直在折腾一件很具体、也很个人的事：

怎么把我平时在 Telegram 里产生的语音、碎片想法、链接和临时感受，稳定地沉淀进 Obsidian，而不是永远散落在聊天记录里。

这个问题看起来不大，但它其实卡了我很久。

真正折磨人的，从来不是“我没有工具”，而是这些很具体的小失控：

  • 语音说完就没了
  • 临时想法过两天找不到
  • 链接和素材堆在聊天框里，根本谈不上整理
  • 就算有 agent，也常常只是“会回复”，不会真正落盘

所以这次我想做的，不是再加一个会聊天的 bot，而是搭一个 独立的日志整理 agent。它更像一个记录员，而不是一个泛用聊天助手。

我想解决的几个真实痛点

1. 输入很多，但沉淀断裂

我平时会在 Telegram 里发很多东西：

  • 一段语音
  • 一个念头
  • 一篇文章链接
  • 一个未来可能写成 blog 的主题

这些内容的共同点是：它们都很容易进入系统，却很难进入知识库。

聊天工具天然适合输入，但并不天然适合沉淀。

2. 语音特别容易“说完就算”

语音是最轻松的输入方式，但也是最容易丢失的输入方式。

问题不在于“没有记录”，而在于它通常：

  • 不可快速检索
  • 不方便二次整理
  • 很难回看和引用
  • 更难直接进入写作流程

所以如果没有一条稳定的“语音 → 文本 → 分类 → 归档”链路，语音最后往往只是情绪消费，而不是知识积累。

3. 日志、素材、草稿、正式内容总是混在一起

我过去最大的一个问题，就是所有东西都很容易堆在一起：

  • 每日日志
  • 随手灵感
  • 素材摘录
  • blog 草稿
  • 最终准备发布的内容

当这些东西没有边界时，知识库最后就会变成一个高级杂物间。

所以这次我很明确地做了一件事：

输入层、整理层、草稿层、发布层必须拆开。

我的目标方案

我最后想要的是这样一条链路：

这里最关键的不是工具名，而是分层：

  • Telegram 负责输入
  • journal agent 负责转写和分流
  • Obsidian 负责沉淀和整理
  • Quartz 负责公开发布

我不想让一个 agent 既负责聊天，又负责记日志，还顺手负责发布内容。那种“全能感”看着很爽，最后往往最乱。

为什么我决定用 Telegram Topic，而不是把一切都塞进同一个聊天入口

一开始我也想过，干脆都放在同一个 Telegram 聊天里算了。

但很快我就意识到，这样上下文会非常脏。

我可能上午在聊仓库结构，下午在丢语音日志，晚上又想让它帮我写 blog。所有东西混在同一个会话里，agent 很容易误判，我自己回看也会非常痛苦。

所以我最后决定：

把 Telegram forum topics 当作不同用途的入口

例如：

  • 主助手：日常聊天、排障、任务处理
  • 日志记录：语音、日记、碎片输入
  • 如果之后需要，还可以再加一个 写作整理 topic

这样一来，topic 本身就是一种语义分区。

我怎么定义这个独立的 journal agent

我给这个 agent 的定位非常克制。

它不是陪聊型 agent，也不是全能助手，而是一个：

记录员 / 整理员 / 输入分流器

它应该负责的是：

  • 收语音
  • 转文本
  • 写文件
  • 摘要整理
  • 必要时输出 blog 草稿

它不应该负责的是：

  • 泛聊天
  • 技术排障
  • OpenClaw 配置解释
  • Quartz 运维
  • 广泛的研究讨论

这一步其实特别重要。

因为一旦职责不清，记录员很快就会变成什么都想答一嘴的“半主助手”，最后写日志也不稳，聊天也不专。

我给 Obsidian 设计的目录边界

为了让这条链路真正可用，我把内容层重新拆成了几块：

对应关系很明确：

  • daily/：当天摘要、反思、行动项
  • inbox/voice/：语音原始转写
  • inbox/clippings/：链接、资料、摘录
  • drafts/blog/：博客草稿
  • publish/public/：准备发布的内容

这套结构的价值在于：

输入先被接住，再被整理，而不是直接扔进正式笔记里。

真正麻烦的地方：不是搭 agent，而是驯服默认行为

做到这里时，我以为事情差不多了。结果真正难啃的部分才刚开始。

第一个坑：Telegram topic 的可见性和 mention 行为

一开始 bot 在 topic 里根本不回。

后来排查下来，我发现问题并不是 topic 路由没写，而是 Telegram 的 bot privacy 和 mention 机制本身就很别扭。调整之后，topic 总算能正常回消息了。

第二个坑：journal agent 没有写权限

我给它单独配了 journal agent，也把 topic 绑定过去了，但它第一次尝试写日志时，发现自己只能看到只读路径。

后来我又做了几轮调整：

  • 给它单独放开 workspace 写权限
  • 最后干脆关闭它的 sandbox，让它直接在主机 workspace 上工作

第三个坑：它总把“日志”理解成 memory/YYYY-MM-DD.md

这才是最顽固的问题。

哪怕我已经：

  • 改了 journal 的规则文件
  • 给日志 topic 加了强制 system prompt
  • 新建了一个全新的测试 topic

系统自己的默认记忆习惯，有时候比你后加的业务规则还顽固。

  • daily/
  • inbox/
  • drafts/
  • publish/

那它把“日志”理解成 daily/ 的概率就会高很多。

说白了，这一步不是在“讲道理”，而是在 改变它所处环境的默认值。

这套方案到底解决了什么问题

到这里，我越来越确定：

这件事的重点不是“我成功配了多少配置项”，而是它确实在解决几个很具体的问题。

1. 让输入真正进入知识系统

不是停留在 Telegram，而是有机会进入：

  • daily
  • voice inbox
  • clippings
  • blog draft

2. 让语音有了可靠归宿

语音不再只是聊天记录里的一段波形，而是可以进入文本、摘要，以及后续写作流程。

3. 让不同类型的内容有明确边界

日志不是素材，素材不是草稿，草稿不是公开内容。

4. 让“记录员 agent”有了明确职责

它不再什么都做，而是专门负责：

  • 收
  • 转
  • 写
  • 整

5. 让发布层和私人内容真正分开

这套工作流最后可以接上 Quartz，但只发布 publish/public/，不会把私人内容一锅端上去。

我最大的感受：真正难的是改“默认值”

这次让我印象最深的一点是：

很多系统看起来都“能用”，但当你真的想把它变成自己的工作流时，你会发现自己其实一直在跟默认值打架。

默认的 memory 习惯、默认的聊天上下文、默认的 topic 行为、默认的发布路径……

这些默认值如果不改，系统就会一直试图把你拉回它熟悉的轨道。

所以真正的工作不是“有没有 agent”，而是：

你有没有把它训练到足够贴近你自己的信息流。

接下来我还会继续验证什么

这套方案还没到“完美稳定”的程度，但已经进入一个非常值得继续打磨的阶段。

我接下来最关注的是：

  • journal 在新的独立 workspace + vault 输出路径下，是否终于能把“记录到日志”稳定写进 daily/
  • 语音能不能顺畅进入 inbox/voice/ 和 daily/
  • 能不能从 daily、clippings、voice 很自然地整理出 blog 草稿

如果这几步都跑顺，那这套系统对我来说就不再只是一个实验，而会真正变成我日常写作和记录的一部分。

结尾

如果要总结这次折腾，我觉得一句话就够了：

我不是在搭一个 bot，我是在搭一条从输入到沉淀、再到输出的个人知识流。

而这条流真正有价值的地方，不在于它多炫，而在于它终于开始解决那些很烦、但也很真实的问题：

  • 我说过的东西别丢
  • 我想过的东西别散
  • 我记下来的东西，最好真的能变成文章

如果一个系统能做到这三点，它就已经不是玩具了。
